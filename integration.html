Simply Salt — Referral Scheduling API

Developer integration guide for the Peterman technician app  ·  v1  ·  July 2026
Live demo (real API, watch the calls): evanwalker55.github.io/simply-salt-partner-demo

What this is

Two HTTPS endpoints: one returns a proposed salt-delivery time for an address, the other books it. Plain JSON with a key header. Both respond in about a second; a 10-second client timeout is plenty.

1 · CheckTech confirms customer name, address, ZIP (pre-filled from the work order). App calls partner_check_availability → gets earliest + latest openings for each of the next four Saturdays.
→
2 · AskApp shows the options in a dropdown. Customer picks one. Nothing is booked yet.
→
3 · BookApp calls partner_book_job with the chosen date/time, phone, and water type. Customer and job are created in Simply Salt's system, tagged to your technician.
Pre-fill customer name, address, ZIP, and phone from the active work order, and take the technician's name from the login session. Done that way, a referral is two taps and the tech types nothing.

1POST /partner_check_availability

Read-only. Returns the earliest and latest open slot for each of the next four Saturdays (open = between 9:00 AM and 4:00 PM starts, with 30 minutes clear on each side of existing deliveries). Every ZIP code is accepted, so you can skip ZIP validation on your side. Render the options in a picker; stops is informational.

If a day is fully booked, earliest and latest are the same time (after the last existing stop) — collapse them to one option in your picker.

2POST /partner_book_job

Creates the customer and the scheduled delivery in Simply Salt's system in one call. Call only after the customer accepts the time.

Reference implementation

// Node/backend example — same two calls the live demo page makes (view its source for a browser version)
const BASE = "https://primary-6xmr-production.up.railway.app/webhook";
const KEY  = process.env.SIMPLY_SALT_PARTNER_KEY;   // production key, server-side only

async function call(path, body) {
  const r = await fetch(`${BASE}/${path}`, {
    method: "POST",
    headers: { "Content-Type": "application/json", "x-partner-key": KEY },
    body: JSON.stringify(body)
  });
  const data = await r.json();
  if (!r.ok) throw new Error(JSON.stringify(data));
  return data;
}

// step 1 — when the tech opens the referral screen
const avail = await call("partner_check_availability",
  { customer_name: job.customerName, address: job.street, zip: job.zip });
// build the picker from avail.options (each has .earliest and .latest)

// step 2 — when the customer picks an option
const booking = await call("partner_book_job", {
  customer_name: job.customerName, phone: job.phone, address: job.street, zip: job.zip,
  scheduled_date: chosenOption.date, scheduled_time: chosenSlot.scheduled_time,
  water_type: "city", referred_by: session.technicianName, partner_company: "Peterman Brothers"
});

Rollout checklist

  1. Build against the demo key — it hits the real scheduler, so times returned are genuine. Tell us before load-testing.
  2. Add the referral tile/button to your existing referral screen; pre-fill fields from the active work order.
  3. Swap in the production key (delivered privately), stored server-side.
  4. Send one live test booking; Simply Salt confirms it landed with the referral note, then you ship.
Need something different (extra fields, technician IDs instead of names, a completion callback for commission reconciliation)? Changes on the Simply Salt side usually ship the same day.
